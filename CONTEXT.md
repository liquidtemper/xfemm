# Context

## Purpose

`xfemm` is the solver-side FEMM-compatible codebase used to mesh, solve, and postprocess magnetic, electrostatic, current-flow, and heat-flow problems outside the Windows FEMM GUI.

Current work in this repo is focused on a new magnetics-only pre-mesh feature:

- Virtual Gap on line segments

The intent is to let a single magnetic line segment stand in for a very small physical nonmagnetic gap, such as `50-100 um`, without forcing the true tiny geometry to be meshed directly.

## Current Feature Status

Virtual Gap is now implemented as a magnetic preprocessor / mesher-side geometry expansion feature.

Scope today:

- planar magnetostatic magnetic problems only
- line segments only
- implemented as geometry rewrite before Triangle meshing
- no new PDE formulation yet

The feature is active through new per-segment fields stored in magnetic `.fem` files:

- `VirtualGapEnabled`
- `VirtualGapPhysical`
- `VirtualGapMesh`

## Architecture

### Segment data model

Main files:

- [`cfemm/libfemm/CSegment.h`](C:\GIT\xfemm\cfemm\libfemm\CSegment.h)
- [`cfemm/libfemm/CSegment.cpp`](C:\GIT\xfemm\cfemm\libfemm\CSegment.cpp)

Magnetic segments now carry:

- on/off state for virtual-gap handling
- physical equivalent gap thickness
- optional explicit ribbon mesh thickness

These fields are part of the segment object itself, not an external side table.

### FEM serialization / deserialization

Main files:

- [`cfemm/libfemm/FemmReader.cpp`](C:\GIT\xfemm\cfemm\libfemm\FemmReader.cpp)
- [`cfemm/libfemm/FemmProblem.cpp`](C:\GIT\xfemm\cfemm\libfemm\FemmProblem.cpp)
- [`cfemm/fmesher/fmesher.cpp`](C:\GIT\xfemm\cfemm\fmesher\fmesher.cpp)

The new segment fields now round-trip in `.fem` files.

Important behavior:

- the input `.fem` can specify a virtual-gap segment directly
- the mesher expands the geometry
- the nonperiodic mesher path writes the rewritten / expanded FEM back out so downstream solver stages see the expanded geometry consistently

### Magnetic poly writer / ribbon expansion

Main file:

- [`cfemm/fmesher/writepoly.cpp`](C:\GIT\xfemm\cfemm\fmesher\writepoly.cpp)

For each enabled magnetic segment, the mesher now:

1. computes a thin ribbon centered on the original line
2. sets ribbon thickness to:
   - `VirtualGapMesh` if provided
   - otherwise a derived thickness based on the physical gap
3. replaces the original single segment with a closed ribbon topology
4. inserts a temporary block label inside the ribbon
5. creates or uses a temporary autogenerated material with:
   - `mu_r = g_mesh / g_phys`
   - `sigma = 0`
   - no magnetization
6. applies local mesh constraints so the ribbon is actually resolved

This is a geometry-equivalent thin-layer approximation, not a special solver term.

### Autogenerated temporary materials

Material creation is currently tied to the mesher rewrite path in:

- [`cfemm/fmesher/writepoly.cpp`](C:\GIT\xfemm\cfemm\fmesher\writepoly.cpp)

Current behavior:

- virtual-gap ribbons use autogenerated material names of the form `__virtual_gap_*`
- these materials are injected only as needed for the expanded geometry

## Guardrails

Current guardrails intentionally reject risky cases before Triangle is asked to mesh them.

Current explicit restrictions:

- non-planar magnetic problems are rejected
- segments using periodic or antiperiodic boundary properties are rejected
- geometries whose offset ribbon would collapse, self-intersect, or conflict with nearby geometry are rejected

This is deliberate. The current implementation is meant to be conservative rather than silently generate bad geometry.

### Case definitions

#### Case 1: steel-to-steel interruption

- U-core style electromagnet
- coil on one leg
- interrupted steel return path on a short internal contact segment
- variants:
  - nominal
  - virtual gap `100 um`
  - true geometry `100 um`

Primary metrics:

- coil flux linkage
- inductance
- average normal `B` through the interrupted steel section

This is the main reference-quality comparison case.

#### Case 2: surface-layer / slot-opening style perturbation

- center-post electromagnet
- pole shoe facing a keeper across a small working air gap
- virtual gap placed at the post-to-shoe interface

Primary metrics:

- average normal `B` across the working-gap line
- peak `|B|` in a sampled working-gap window
- coil inductance / flux linkage

Important metric note:

- the peak-window metric was narrowed so it samples the actual working air region
- earlier versions were accidentally sampling a virtual-ribbon hotspot instead of the physically relevant air region

#### Case 3: PM-to-steel interface separation

- permanent magnet closing the bottom opening of a steel U-frame return path
- virtual gap placed just inside the magnet-side interface region
- no explicit true-gap reference retained for this case

Primary metrics:

- average normal `B` through the top steel bridge
- local `|B|` probes in the return path / cavity

This geometry was simplified several times during the sprint because earlier PM demo variants were too weak or topologically fragile.

### Case 1

Current results:

- nominal inductance: `0.327956`
- virtual-gap inductance: `0.317721`
- true-gap inductance: `0.318812`

Observed change:

- nominal -> virtual gap inductance: about `-3.12%`
- virtual-gap vs true-gap inductance error: about `0.34%`

This is the strongest evidence that the current virtual-gap approximation is behaving as intended.

### Case 2

Current results:

- line-average `B`: `0.00289284 -> 0.0025891`
- peak working-region `|B|`: `0.00356001 -> 0.00314031`

Observed change:

- line-average `B`: about `-10.5%`
- peak sampled working-gap `|B|`: about `-11.8%`
- local probes: about `-10.9%` and `-11.0%`

Inductance change is intentionally small in this case. The dominant visible effect is the local field reduction near the surface layer.

### Case 3

Current results:

- bridge line-average `B`: `1.14744 -> 0.995146`
- local probe 0: `0.883786 -> 0.743185`
- local probe 1: `0.679456 -> 0.586718`

Observed change:

- bridge line-average `B`: about `-13.3%`
- probe 0: about `-15.9%`
- probe 1: about `-13.6%`

This is now a clear enough delta for a demo case, even though it is not paired with a true-geometry reference model.

## Known Constraints

- feature is magnetics-only today
- planar only
- line segments only
- periodic / antiperiodic boundary segments are intentionally unsupported
- the demo workflow depends on:
  - `fmesher`
  - `fsolver`
  - `fpproc-vgap`
  - Python with `matplotlib`

Important practical note:

- case 1 carries the explicit true-gap validation target
- case 2 and case 3 are intended primarily as clear delta demos, not direct truth-model accuracy studies

## Re-entry Guidance

If Virtual Gap needs more work, use this order:

1. check segment serialization first
2. inspect the rewritten `.fem` generated by the mesher
3. verify the autogenerated `__virtual_gap_*` material properties
4. inspect local mesh constraints on the ribbon
5. check whether the ribbon is too close to existing geometry
6. rerun the benchmark harness and compare summary deltas

If a demo case becomes unstable, the first things to check are:

- unlabeled enclosed air pockets
- duplicate / near-duplicate vertices created by the ribbon expansion
- field probes or peak windows accidentally sampling the ribbon itself instead of the intended physical region

## Non-Goals For Re-entry

- do not add a new solver formulation unless the geometry-equivalent ribbon approach proves insufficient
- do not weaken the current guardrails just to force unsupported cases through
- do not judge case quality only by visual plots; keep a scalar metric delta in the summary
